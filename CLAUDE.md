# 開發原則

## 架構與設計（規劃時嚴謹）
- **架構優先**: 此專案以「做大」為前提開發，規劃階段不採 MVP 思維。每個跨模組、跨服務、跨資料源的設計決定，都要先評估「之後規模成長 10x / 100x 時是否需要重寫」。寧可前期多花時間規劃對，也不要留下難以回頭的耦合。
- **資料邊界清楚**: 與外部系統（如 CMS Firebase）整合時，務必同步資料到自己的 Firestore，並定義自己的 schema 與轉換層。禁止前端直接讀外部資料源。
- **抽象層保留彈性**: 素材 URL、影片來源、第三方服務等可能變動的依賴，要以欄位/介面包裝，避免硬編碼具體實作。
- **可擴展性**: 設計同步、渲染、批次處理等任務時，預設要能水平擴展（idempotent、可重試）。

## 程式碼品質（實作時敏捷）
- **快速實作優先**: 規劃要嚴謹，但實作階段以「快速產出可測試的版本」為主，不過度工程化。功能能跑、能測，就先 commit；細節優化等需求穩定後再做。
- **錯誤處理**: 先當程式碼會順利運行，不寫一堆錯誤處理。真的有需要，將 error message 印出即可，不用針對個別錯誤訊息處理。關鍵路徑（金流、寫入、外部 API）才需要明確處理失敗。
- **加上簡明的註解**: 為了讓他人知道新增的程式碼的目的及用途，請簡單明瞭地以註解說明。
- **Convention over Configuration**: 應遵循框架和工具的官方慣例與最佳實務，避免過度客製化配置。尊重官方文件建議的檔案結構、命名規範和開發模式。
- **官方文件導向**: 為了避免使用過時的資訊，請使用 context7 MCP 搜尋符合當前版本的文件，基於官方文件內容開發，而不是依賴舊的訓練資料。
- **最新穩定版**: 新增套件時，請確認使用的是最新穩定版本號。
- **TDD for API**: 開發 API 時，嚴格遵循測試驅動開發（TDD）流程。先編寫失敗的測試，提交後，再編寫能讓測試通過的程式碼。

## 工作流程
- **規劃優先**: 始終先規劃，後編碼。在撰寫任何程式碼前，必須先提出詳細的步驟計畫並獲得批准。架構層級的變更需另外列出影響範圍與權衡分析。
- **聚焦指令**: 請聚焦在我的指令範圍，如果你覺得有指令以外的部分也需要修改，請先提出，經過我允許再實作。
- **專注於當前開發的功能**: 不應在修改程式碼的時候刪除、修改與此次任務無關的程式碼或註解。
- **技術債管理**: 更改實作機制時，務必刪除舊方法！不要為了向前相容而保留！不然會一直留下技術債！
- **git mv 取代 mv 指令**: 改名時，應該用 git mv 而不是 mv 指令。
- **自動 build 前端**: 每當編輯了 frontend 的程式碼，請記得執行 `npm run build`。
- **以 `npm run deploy` 進行 deploy**: 這個指令會一併進行 build，避免忘記 build。

## 文件同步
- **即時更新 SPEC**: 每當進行修改、新增、移除功能，或是調整架構，請務必同步更新 @doc/spec.md。
- **更新 todo.md**: 每當完成一項任務並 commit 後，到 @doc/todo.md 找尋對應的事項，標記為完成、並移到 DONE 區塊後，以簡單的一句話說明成果，並以括號附註 commit id。例如： `[x] Improve: tts 有時候會漏掉數字，要在 prompt 中提醒 Gemini 不要漏掉數字 \n\t=> 撰寫驗證腳本並修改 SKILL.md 的指令，並修改 SKILL.md 的結構以符合規範 (6aa24b4)`。但在改完後不用再 commit 一次。
