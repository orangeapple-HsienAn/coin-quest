## §4 簽到頁面 — 命運卡（風險事件）

### 需求書描述
§4.4 步驟三：「系統進行隨機判定，決定觸發風險事件、無事發生，或正面獎勵」「顯示事件插圖、描述文案與損益金額」

### ⚠️ 缺乏的具體規格

需求書**沒有提供命運卡完整清單**。每張命運卡需定義：

| 欄位 | 說明 | 範例 |
|---|---|---|
| `title` | 卡片標題 | 「上班太累導致精神不濟」 |
| `description` | 事件描述文案 | 「為了補充體力買了一堆補品」 |
| `type` | 類型 | `risk` / `neutral` / `reward` |
| `coinLoss` | 風險損失金額（type=risk 時） | 3000 |
| `coinGain` | 獎勵金額（type=reward 時） | 2000 |
| `category` | 對應的保險類別（type=risk 時可用） | `medical` / `accident` / `home` / `theft` / `none` |
| `imageUrl` | 事件插圖 URL | （目前空字串） |
| `triggerWeight` | 抽中機率權重 | 1 |

### 現況
- ✅ 工程已建立 seed 資料（`functions/src/seed/fateCards.ts`），含風險、中性、獎勵共 N 張
- ❌ 卡片內容、金額、機率分布**未經 PM review**
- ❌ 事件插圖（`imageUrl`）尚未提供

### 需 PM 提供
1. **完整命運卡清單**（建議 risk:neutral:reward ≈ 30%:40%:30%）
2. **每張卡片的金額** 是否合理（過大會破壞經濟平衡）
3. **每張 risk 卡片對應的保險類別**（決定哪張保險可以理賠）
4. **事件插圖**（4 種類別 × N 張，可請設計師統一風格）

## §10 保險頁面 — 保險卡片

### 需求書描述
§10.2：「以卡片網格排列顯示所有可購買或持有的保險產品」「產品標頭、理賠金額、保障說明」
§10.4：「系統檢查使用者是否持有對應事件類型且效期內的保險」

### ⚠️ 缺乏的具體規格

需求書**沒有提供保險產品完整清單**，也**沒有定義「對應事件類型」的對應關係**。

每項保險產品需定義：

| 欄位 | 說明 | 範例 |
|---|---|---|
| `name` | 保險名稱 | 意外險、醫療險、住宅火災險、癌症險、竊盜險 |
| `icon` | Icon emoji 或圖示 | 🩹 / 🏥 / 🏠 / 🎗️ / 🔒 |
| `description` | 保障說明文字 | 「須符合外來、非疾病、突發的三大意外特性」 |
| `category` | 可抵銷的命運卡類別 | `accident` / `medical` / `home` / `cancer` / `theft` |
| `claimAmount` | 最高理賠金額 | 5000 |
| `premium` | 購買費用（金幣） | 800 |
| `durationDays` | 效期（天） | 30 |

### 現況
- ✅ 工程已建立 seed 資料（`functions/src/seed/insuranceProducts.ts`），共 5 項保險
- ✅ 已實作命運卡與保險的對應邏輯（依 `category` 配對）
- ❌ 5 項保險的金額 / 費用 / 效期**未經 PM 確認合理性**
- ❌ 保險 ↔ 命運卡的對應關係**未經 PM review**

### 需 PM 提供 / 確認
1. **保險產品清單**（目前 5 項是否齊全？有無遺漏類別？）
2. **理賠金額 vs 保費 vs 命運卡損失金額** 三者數值是否合理
3. **效期長度**（30 天合理嗎？要不要差異化？）
4. **保險與命運卡的 category mapping 表**（決定哪張保險救哪張卡）

## §3.1 首頁任務 — 每日測驗（2026/05 更新規格）

### 需求書最新描述
> - 每日測驗：點擊後開始進行測驗，直至完成，每日只能一次
> - **所有「小測驗」和「小議題測驗」題目作為題庫**，每日測驗從題庫中抓取 **5 題**
> - 每一題需標註是**第幾課第幾單元**的題目
> - 若是學生**沒有上過的內容**，需於詳解處標註，讓學生知道「後面才會上到」
> - 測驗後需有**鼓勵**，且每答對一題 **+20 金幣**
> - 基礎完成 **+200 金幣**

### 獎勵公式（已確認）
- **基礎 200**（完成即得）+ **答對數 × 20**（最高 5 題 = +100）
- → 範圍 **200 ~ 300 金幣**

### 「未上過」判斷邏輯（已確認）
- 依該 unit 是否「**解鎖**」（用戶等級達門檻 + 前一課完成）
- ⚠️ **依賴 A3 解鎖機制**，A3 未完成前此提示無法生效

### 實作現況
- ❌ 目前「每日測驗」只是 placeholder（HomePage tile 點擊導去 /course，無實際測驗）
- ❌ 題庫聚合器、抽題、出處標註、鼓勵、計分 全部未實作
- 📋 已排入 M1（預估 +2.6 day）

### 待補規格
- 題庫不足 5 題時的處理（早期只有 lesson-1 時題目可能 < 5）
- 鼓勵文案內容（全對 / 過半 / 不及格 三種）
- 每日重置時間（建議統一台北 00:00）

## §5 章節金幣領取規則（2026/05 補充）

### 規則（已確認）
| 章節類型 | 金幣領取 | 經驗值 |
|---|---|---|
| 小故事 | — （本來就不給金幣） | 1 次 |
| 小知識 | — （本來就不給金幣） | 1 次 |
| 小測驗 | **1 次**（答對×10） | 1 次 |
| 小議題 | — | 1 次 |
| 小議題測驗 | **1 次** | 1 次 |
| **小遊戲** | **前 3 次遊玩，每次依當次星數發**（50/100/150） | 無 |

### 小遊戲「3 次」語意（已確認）
- 重玩前 3 次，**每次依當次星數獨立發金幣**
- 例：第 1 次 1★ +50、第 2 次 3★ +150、第 3 次 2★ +100 → 共領 300
- 第 4 次起遊玩不再發金幣（仍可玩）

### 實作現況
- ✅ 小故事/知識/測驗：現有 `completeChapter` 已用 `completedChapters` 擋重複（金幣+經驗都只一次）
- ❌ 小遊戲：現有邏輯也擋重複，**需改成允許前 3 次發金幣**
- 📋 需改：`functions/src/api/course.ts`（後端計數）+ 前端傳 `chapterType` + GameResultCard 顯示剩餘次數
